# issule-verify — 结果验证（人机契约履行）

**适用于任何类型任务。以 issule-goal 定义的三重验证退出条件为唯一锚点。**
**人只做一件事：执行 T3 步骤并回报结果。**

遵循 [output-protocol.md](./output-protocol.md) LAW1。

---

## 调用方式

```
/issule-verify #N        # 验收指定 issue（默认 talkflow repo）
/issule-verify #N <repo> # 指定仓库
```

---

## 执行协议

### Phase 1 — 读取契约

读取 issue body + 所有 comments。

提取「三重验证退出条件」：
- T1 数字化命令 + 期望值
- T2 AI推理判断标准 + 置信度阈值
- T3 人工步骤 + 验收标准

**契约是唯一锚点，不接受其他来源的验收标准。**

---

### Phase 2 — 执行 T1（agent 自动）

逐条运行命令，对比期望值：

```
✅ [条件描述] → 实际=[X]，期望=[Y]
❌ [条件描述] → 实际=[X]，期望=[Y]
   修复方向：[一句话]
```

有 ❌ → 记录，继续执行其余条件，最后汇总。

---

### Phase 3 — 执行 T2（agent 自判）

取样本（5-10条），应用判断标准，输出：

```
T2 [条件描述]
   置信度：XX%
   样本：[样本1] / [样本2] / ...
   判定：通过 / 不通过 — [一句话理由]
```

置信度 < 60% → 自动降级为 T3，给人精确步骤。

---

### Phase 4 — 给人 T3（人执行，agent 等待）

**给步骤，不给判断题。人只需要执行，不需要思考。**

```
👤 T3 [条件描述]（预计 N 秒）

第1步：[精确操作]
第2步：[看什么 / 找什么]
预期：[符合 → 回复"✅"；不符合 → 描述问题]
```

等人回报后，继续 Phase 5。

---

### Phase 5 — 输出验收报告，POST 到 Issue

**必须以 YES 或 NO 开头。**

判定规则：
```
YES → T1 全部通过 + T2 置信度均值 ≥ 阈值 + T3 全部通过
 NO → 任何 T1 失败 / T2 均值未达阈值 / T3 未通过
```

报告格式：

```markdown
## 验收报告：#N [issue 标题]

# ✅ YES — 可交付
（或）
# ❌ NO — [N] 项未通过，不可交付

执行时间：[timestamp]

---

T1 数字化（X/Y 通过）
✅ [条件] → 实际=[X]，期望=[Y]
❌ [条件] → 实际=[X]，期望=[Y] → 修复：[一句话]

T2 AI推理
[条件]（置信度 XX%）
样本：[样本1] / [样本2]
判定：通过 / 不通过 — [理由]

T3 人工
[条件] → [人回报的结果]

---

❌ 未通过项修复清单

| 条件 | 实际 | 期望 | 修复方向 |
|------|------|------|---------|
| [条件] | [X] | [Y] | [方向] |
```

POST 到 issue comment：

```bash
gh api repos/$REPO/issues/$N/comments \
  -X POST -f body="$REPORT"
```

---

## 行为准则

| 做 | 不做 |
|----|------|
| 以 issue 里的三重条件为唯一锚点 | 重新设计验收标准 |
| T1 全部跑完再汇总 | 发现第一个 ❌ 就停 |
| T3 给精确步骤，等人回报 | 替人猜测 T3 结果 |
| 置信度 < 60% 的 T2 升级为 T3 | 低置信度强行输出判定 |
| YES/NO 开头，结论先行 | 把结论埋在报告末尾 |
| 有 ❌ 给修复方向，不只报告问题 | 报告完就结束 |
| 全部通过才说 YES | T3 未完成就输出 YES |
